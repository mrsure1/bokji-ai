# 복지AI 프로젝트 계획서

> **제품명:** 복지AI (bokji-ai) — 맞춤형 복지 혜택 알리미
> **한 줄 정의:** 행정 용어를 몰라도 자연어로 상황을 말하면, 받을 가능성이 있는 공공 복지 혜택을 쉬운 말로 설명하고 마감 임박 혜택을 선제 안내하는 AI 복지 상담·알림 서비스
> **기준일:** 2026-06-09 / **문서 출처:** PRD·TRD 기반

---

## 1. 프로젝트 목표

| 구분 | 목표 | 세부 내용 |
| --- | --- | --- |
| 핵심 가치 | 복지 통역사 | 정부24·복지로·지자체 공고의 어려운 문장을 즉시 이해 가능한 생활 언어로 변환 |
| 기능 목표 | 자연어 복지 상담 | "퇴사해서 월세가 부담돼" 같은 일상어로 관련 혜택을 찾아줌 |
| 기능 목표 | 쉬운 말 설명 | 중위소득·소득인정액·위기사유 등 행정 용어를 생활 언어로 풀이 |
| 기능 목표 | 개인 조건 기반 추천 | 지역·나이·가구·소득·주거·직업을 반영해 "가능성 높음 / 확인 필요 / 가능성 낮음"으로 구분 |
| 기능 목표 | 선제 알림 | 새 혜택 등록·마감 임박 시 앱을 다시 열지 않아도 문자·이메일로 안내 |
| 기능 목표 | 신청 행동 유도 | 신청 기한·준비 서류·신청 링크·문의처를 명확히 제시 |
| 기술 목표 | 직관적 시각화 UX | 복잡한 복지 정보를 모바일에서 한눈에 파악하는 PWA 화면 설계 |
| 기술 목표 | 내부 데이터 기반 | 사용자 요청 시점에 외부 공공 API를 직접 호출하지 않고 내부 DB·벡터 검색 사용 |
| 안전 원칙 | 수급 미확정 표현 | "반드시 받을 수 있다"가 아니라 "신청 가능성이 있다 / 조건 확인 필요"로 안내 |

## 2. 기술 스택

| 영역 | 기술 | 비고 (구현 확정 사항) |
| --- | --- | --- |
| 프레임워크 | Next.js 16 (App Router) + React 19 | Route Handler로 백엔드 구성 |
| 언어 | TypeScript 5 | |

| 앱 형태 | PWA(목표) | 현재 일반 웹앱 — 매니페스트/서비스워커 미적용(예정) |

| 벡터 검색 | pgvector | 확장 활성화·`benefit_embeddings` 테이블 존재, 임베딩 생성·검색은 예정 |
| 인증 | Supabase Auth (`@supabase/ssr`) | 쿠키 세션 스캐폴딩만 존재, 로그인 UI/플로우 예정 |

| 스케줄러 | Vercel Cron | `POST /api/cron/collect`, `CRON_SECRET` Bearer 인증 |

| 이메일 알림 | Resend (Email) | 키 설정만 존재, 발송 로직 예정 |
| 테스트 | Playwright (E2E) | |

## 3. 서비스 흐름도 (아키텍처)

### 3.1. 전체 데이터 흐름

```text
공공데이터 API / 복지 사이트
  (복지로 중앙부처 · 복지로 지자체 · 정부24)
        │
        ▼
  데이터 수집 워커 (일 1회, 소스별 어댑터)
        │
        ▼
  정규화 / 중복 제거 / facet 분류
        │
        ├───────────────────────┐
        ▼                       ▼
   PostgreSQL              Embedding 생성
   (benefits 테이블)        (pgvector)
        │                       │
        └───────────┬───────────┘
                    ▼
        하드필터 → 하이브리드 검색(전문검색+벡터)
                    │
                    ▼
   Next.js PWA  ◄─►  API 서버  ◄─►  AI 상담/요약 모듈(Gemini)
                    │
                    ▼
            SMS / Email 알림 워커
```

### 3.2. 추천(RAG) 처리 단계

| 단계 | 처리 내용 |
| --- | --- |
| ① 입력 | 사용자가 자연어로 상황 입력 |
| ② 의도 추출 | AI가 의도·생활 상황·필요 지원 분야 추출 |
| ③ 구조화 필터 | 프로필 + 추출 의도로 지역·생애주기·가구·주제 하드필터 적용 |
| ④ 벡터 검색 | 후보군 내에서 의미적으로 가까운 혜택 검색 |
| ⑤ 점수화·분류 | "가능성 높음 / 확인 필요 / 가능성 낮음"으로 분류 |
| ⑥ 쉬운 말 설명 | AI가 후보 혜택을 생활 언어로 설명 |
| ⑦ 추가 질문 | 확정 판단이 필요한 조건은 한 번에 하나씩 추가 질문 |

## 4. 기능

| 기능 | 설명 | 우선순위 | 구현 현황 (2026-06-09) |
| --- | --- | --- | --- |
| 사용자 프로필·온보딩 | 맞춤 추천에 필요한 정보를 단계적으로 입력 | High | DB 스키마만 존재, 저장·연동 예정 |
| 자연어 복지 상담 | 일상어 입력 → 의도 해석 → 혜택 카드 추천 | High | 동작(Gemini), 하이브리드 검색으로 고도화 중 |
| AI 쉬운 말 요약 | 공고 원문을 한 줄 요약·대상·금액·기한 등으로 변환 | High | 읽기 경로 존재, 생성 파이프라인 예정 |
| 문자/이메일 알림 | 새 맞춤 혜택·마감 임박 시 선제 발송 | High | 스키마·env 존재, 발송 로직 예정 |
| 혜택 저장·체크리스트 | 관심 혜택 저장 + 준비 서류·절차 체크 | Medium | 스키마 존재, DB 저장 예정 |
| 관리자 검수 | 수집 공고·AI 요약·추천 태그·발송 상태 검수 | Medium | `review_status` 필드만 존재 |
| 데이터 수집 파이프라인 | 공공 API 일 1회 수집·정규화·facet 태깅 | High | **구현 완료 — 15,971건 적재** |

## 5. 사용 기술 (기능별 적용 기술)

| 기능 | 사용 기술 |
| --- | --- |
| 자연어 복지 상담 | Google Gemini (JSON 스키마 강제 출력), Next.js Route Handler |
| 쉬운 말 요약 | Google Gemini, 백그라운드 요약 워커(예정) |
| 개인 조건 추천 | PostgreSQL facet 하드필터 + pgvector 벡터 검색 + 하이브리드 검색(RRF, 예정) |
| 한국어 전문검색 | Postgres FTS, pgroonga/pg_bigm 확장 도입 검토 중 |
| 데이터 수집 | 소스별 어댑터(`src/lib/collector/*`), `pg` 직접 연결, Vercel Cron |
| 데이터 출처 | 복지로 중앙부처(448) · 복지로 지자체(4,561) · 정부24(10,962) = **15,971건**, SSIS 공통코드(보조 온톨로지) |
| 프로필·인증 | Supabase Auth(`@supabase/ssr`) 쿠키 세션 |
| 문자/이메일 알림 | Solapi(SMS), Resend(Email) |
| UI/화면 | Next.js 16 + React 19, Tailwind CSS v4, PWA(예정) |
| 데이터 저장 | Supabase PostgreSQL + pgvector, 7개 테이블 + facet 컬럼/인덱스 |

## 6. 기대 효과

| 관점 | 기대 효과 |
| --- | --- |
| 정보 접근성 | 행정 용어를 몰라도 자연어 한마디로 받을 수 있는 혜택을 찾아, 정보 격차 해소 |
| 신청 포기 감소 | 어려운 용어를 쉬운 말로 풀어주고 신청 절차를 안내해 복지 신청 포기율 감소 |
| 놓침 방지 | 마감 임박·신규 혜택을 선제 알림으로 안내해 "몰라서 못 받는" 사례 감소 |
| 시간 절약 | 흩어진 정부24·복지로·지자체 공고 15,971건을 한 곳에서 통합 검색 |
| 취약계층 포용 | 큰 글씨·짧은 문장·명확한 버튼으로 시니어·정보 취약계층 접근성 향상 |
| 행동 전환 | 단순 정보 제공을 넘어 신청 기한·서류·링크 제시로 실제 신청 행동까지 연결 |
| 안전성 | 수급 확정이 아닌 "가능성" 표현으로 잘못된 기대·오안내 위험 최소화 |
| 운영 안정성 | 외부 API 장애 시에도 내부 캐시 데이터로 서비스 지속 가능 |
