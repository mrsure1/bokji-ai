# [제품 요구사항 정의서] PRD (Product Requirement Document)

## 1. 프로젝트 개요 (Overview)

* **제품명:** 우리 동네 맞춤형 복지 혜택 알리미 (가칭: WelfareFit)
* **제품 정의:** 사용자가 행정 용어를 몰라도 자신의 상황을 자연어로 말하면, 공공 복지 정보를 기반으로 받을 가능성이 있는 혜택을 쉬운 말로 설명하고, 신청 가능하거나 마감이 임박한 혜택을 문자 또는 이메일로 선제 안내하는 AI 복지 상담 및 알림 서비스
* **핵심 포인트:** 단순 복지 검색이 아니라, 정부24/복지로/지자체 복지 공고의 어려운 문장을 사용자가 즉시 이해할 수 있는 생활 언어로 바꿔 주는 "복지 통역사 + 개인 알림 비서" 역할을 한다.
* **타겟 유저:**
    * 정부 혜택을 직접 검색할 시간이나 경험이 부족한 청년, 구직자, 사회초년생
    * 어려운 행정 용어 때문에 복지 신청을 포기하는 정보 취약계층 및 시니어
    * 갑작스러운 실직, 질병, 주거비 부담, 육아, 돌봄 등 생활 변화로 지원 제도를 찾아야 하는 사용자

## 2. 핵심 가치 및 목표 (Goals)

* **자연어 복지 상담:** 사용자가 "퇴사해서 월세가 부담돼", "부모님 병원비가 걱정돼"처럼 일상어로 말해도 관련 복지 혜택을 찾아준다.
* **쉬운 말 설명:** "중위소득", "소득인정액", "위기사유", "수급자격" 같은 행정 용어를 생활 언어로 풀어 설명한다.
* **개인 조건 기반 추천:** 사용자의 지역, 나이, 가구 상황, 소득 구간, 주거 상태, 직업 상태 등을 반영하여 혜택을 "가능성 높음 / 확인 필요 / 가능성 낮음"으로 구분한다.
* **선제 알림:** 새 혜택이 등록되거나 신청 마감이 가까워지면 사용자가 앱을 다시 열지 않아도 문자 또는 이메일로 알려준다.
* **신청 행동 유도:** 단순 정보 제공을 넘어 신청 기한, 준비 서류, 신청 링크, 문의처를 명확히 보여줘 사용자가 다음 행동을 바로 할 수 있게 한다.

## 3. 핵심 기능 요구사항 (Functional Requirements)

### 3.1. 사용자 프로필 및 온보딩 (User Profile)

* **요구사항:** 사용자는 회원가입 또는 첫 상담 시작 시 맞춤 추천에 필요한 정보를 입력할 수 있어야 한다.
* **필수 입력 정보:**
    * 출생년도 또는 연령대
    * 거주지 시/도, 시/군/구
    * 현재 상태: 학생, 구직자, 직장인, 프리랜서, 소상공인, 무직, 은퇴자 등
    * 알림 수신 채널: 문자, 이메일 중 하나 이상
* **선택 입력 정보:**
    * 가구원 수 및 동거 가족 구성: 1인 가구, 부부, 자녀 있음, 부모 부양, 조손 가구 등
    * 소득 구간: 정확한 금액이 부담스러운 사용자를 위해 "소득 없음", "낮음", "중간", "잘 모르겠음" 같은 선택지를 제공한다.
    * 주거 형태: 월세, 전세, 자가, 고시원/기숙사, 임시 거주 등
    * 고용 상태 상세: 퇴사 예정, 최근 실직, 구직 중, 휴직, 폐업, 일용직 등
    * 관심 분야: 생계비, 주거비, 의료비, 교육비, 육아, 장애, 노인, 청년, 창업/소상공인 등
    * 특수 상황: 임신/출산, 장애, 질병/입원, 한부모, 다문화, 보훈, 재난 피해 등
* **UX 원칙:** 민감한 정보는 한 번에 강제하지 않고, 상담 흐름 중 필요한 시점에 추가 질문으로 받는다.
* **우선순위:** High

### 3.2. 자연어 복지 상담 (Conversational Welfare Matching)

* **요구사항:** 사용자가 일상적인 문장으로 현재 상황을 입력하면, 시스템은 의도를 해석하고 관련 복지 혜택을 추천해야 한다.
* **예시 입력:** "이번 달에 퇴사해서 생활비가 부족해", "월세 지원 받을 수 있어?", "부모님 병원비가 너무 많이 나와"
* **응답 형식:**
    * 상황 요약: 사용자의 말을 쉬운 표현으로 정리
    * 추천 혜택 카드: 혜택명, 받을 수 있는 내용, 신청 조건, 신청 방법, 마감일
    * 적합도 표시: 가능성 높음 / 확인 필요 / 가능성 낮음
    * 추가 질문: 더 정확한 추천에 필요한 정보만 짧게 질문
* **우선순위:** High

### 3.3. AI 행정 문서 쉬운 말 요약 (Plain-Language Summarization)

* **요구사항:** 사용자가 공고를 클릭했을 때 원문의 복잡한 줄글 대신 쉬운 요약을 최상단에 보여주어야 한다.
* **요약 항목:**
    * 한 줄 요약
    * 누가 받을 수 있는지
    * 얼마 또는 무엇을 받을 수 있는지
    * 언제까지 신청해야 하는지
    * 어디서 어떻게 신청하는지
    * 준비해야 할 서류
    * 헷갈릴 수 있는 행정 용어 풀이
* **표현 원칙:** 초등학생도 이해할 수 있는 짧은 문장으로 작성하고, 원문에 없는 수급 확정 표현은 사용하지 않는다.
* **우선순위:** High

### 3.4. 맞춤형 문자/이메일 알림 (Smart SMS/Email Notification)

* **요구사항:** 시스템은 매일 동기화된 복지 공고 중 사용자 조건과 맞는 항목이 있거나 신청 마감이 가까운 항목이 있으면 문자 또는 이메일로 알려야 한다.
* **알림 유형:**
    * 새로 등록된 맞춤 혜택
    * 신청 마감 임박 혜택
    * 사용자가 저장한 혜택의 변경 사항
    * 추가 정보 입력 시 더 정확히 확인 가능한 혜택
* **알림 내용:** 혜택명, 쉬운 한 줄 설명, 신청 마감일, 앱 상세 화면 링크를 포함한다.
* **수신 관리:** 사용자는 알림 채널, 관심 분야, 알림 빈도, 수신 거부를 직접 설정할 수 있어야 한다.
* **우선순위:** High

### 3.5. 혜택 저장 및 신청 준비 체크리스트 (Saved Benefits)

* **요구사항:** 사용자는 관심 있는 혜택을 저장하고, 신청 전 필요한 서류와 절차를 체크리스트로 확인할 수 있어야 한다.
* **주요 기능:**
    * 관심 혜택 저장
    * 신청 마감일 리마인드
    * 준비 서류 체크
    * 신청 링크 또는 문의처 바로가기
* **우선순위:** Medium

### 3.6. 관리자 검수 화면 (Admin Review)

* **요구사항:** 운영자는 수집된 복지 공고, AI 요약, 추천 태그, 알림 발송 상태를 확인하고 수정할 수 있어야 한다.
* **필요성:** 복지 정보는 사용자 생활에 직접 영향을 주므로 AI 요약과 추천을 운영자가 검수할 수 있는 장치가 필요하다.
* **우선순위:** Medium

## 4. 비기능적 요구사항 (Non-Functional Requirements)

* **사용성(UX):** 모바일에서 읽기 쉬운 PWA 화면을 제공하고, 큰 글씨 모드와 쉬운 용어 설명을 기본 제공한다.
* **성능(Latency):** 일반적인 자연어 상담 응답은 3초 이내를 목표로 하며, 외부 공공 API를 사용자 요청 시점에 직접 호출하지 않는다.
* **정확성:** 혜택 추천은 "수급 확정"이 아니라 "신청 가능성이 있는 혜택"으로 표현한다.
* **보안:** 개인정보와 민감정보는 암호화 저장하며, 맞춤 추천과 알림 목적 외에는 사용하지 않는다.
* **동의:** 문자/이메일 알림, 개인정보 저장, 민감정보 입력은 각각 명확한 동의를 받아야 한다.
* **접근성:** 시니어와 정보 취약계층을 고려하여 충분한 대비, 큰 터치 영역, 짧은 문장, 명확한 버튼 문구를 사용한다.
* **운영성:** 공공 API 변경, 수집 실패, AI 요약 실패가 발생해도 기존 저장 데이터를 기반으로 서비스가 유지되어야 한다.

## 5. MVP 범위 (Initial Release Scope)

* PWA 기반 웹앱
* 사용자 회원가입 및 확장 프로필 입력
* 자연어 복지 상담
* 복지 혜택 카드 추천
* 쉬운 말 요약
* 관심 혜택 저장
* 문자 또는 이메일 알림
* 관리자용 공고/요약 검수 화면

## 6. 제외 범위 (Out of Scope for MVP)

* 사용자를 대신한 복지 신청 대행
* 주민등록, 소득, 금융 정보의 자동 조회
* 수급 가능 여부의 법적 확정 판정
* 카카오 알림톡 연동
* 네이티브 모바일 앱 출시
