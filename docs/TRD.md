# [기술 요구사항 정의서] TRD (Technical Requirement Document)

## 1. 기술 목표

본 서비스는 사용자가 자연어로 자신의 상황을 입력하면, 내부에 수집된 복지 공고 데이터를 기반으로 관련 혜택을 분석하여 모바일 환경에서 가장 보기 좋고 직관적으로 한눈에 파악할 수 있는 UI를 통해 쉬운 말로 설명해 주는 PWA 기반 웹앱이다. (사용자 조건에 맞는 신규 혜택이나 마감 임박 혜택을 알려주는 문자 및 이메일 알림은 사용자가 선택할 수 있는 부수적인 편의 옵션으로 제공된다.)

핵심 기술 목표는 다음과 같다.

* 복잡한 복지 혜택 정보를 화면상에서 보기 좋고 한눈에 알아보기 쉽게 전달하는 직관적인 시각적 사용자 경험(UX)을 설계한다.
* 공공 복지 데이터를 주기적으로 수집하고 내부 DB에 저장한다.
* 복잡한 행정 문서를 쉬운 말 요약 데이터로 변환해 저장한다.
* 사용자의 프로필 조건과 자연어 질문을 함께 활용해 복지 혜택을 추천한다.
* 사용자 요청 시점에는 외부 공공 API를 직접 호출하지 않고 내부 DB와 벡터 검색을 사용한다.
* 부수적 옵션으로서의 문자와 이메일 알림을 안정적으로 발송하고 수신 동의/거부 상태를 관리한다.

## 2. 권장 기술 스택

### 2.1. 프론트엔드

* **프레임워크:** Next.js
* **앱 형태:** PWA(Progressive Web App)
* **언어:** TypeScript
* **UI:** Tailwind CSS 또는 shadcn/ui 기반 컴포넌트
* **주요 화면:**
    * 온보딩/프로필 입력
    * 자연어 상담 채팅
    * 추천 혜택 카드 목록
    * 혜택 상세/쉬운 말 요약
    * 저장한 혜택
    * 알림 설정
    * 관리자 검수 화면

### 2.2. 백엔드

* **런타임:** Next.js Route Handler 또는 별도 Node.js API 서버
* **DB:** PostgreSQL
* **벡터 검색:** pgvector
* **인증:** Auth.js 또는 Supabase Auth
* **파일/로그 저장:** 초기에는 DB 중심으로 관리하고, 추후 객체 스토리지 연동
* **작업 스케줄러:** Vercel Cron, GitHub Actions, Cloudflare Cron, 또는 별도 워커

### 2.3. AI

* **LLM 사용 목적:**
    * 복지 공고 쉬운 말 요약
    * 행정 용어 풀이
    * 사용자 자연어 질문 의도 추출
    * 추천 결과 설명 문장 생성
* **Embedding 사용 목적:**
    * 복지 공고 원문 및 요약문 벡터화
    * 사용자 질문과 의미적으로 가까운 복지 공고 검색
* **중요 원칙:** AI는 수급 가능 여부를 확정하지 않고, 후보 혜택을 설명하고 추가 확인이 필요한 조건을 안내한다.

### 2.4. 알림

* **문자 발송:** 국내 SMS 발송 API 연동
* **이메일 발송:** SMTP 또는 이메일 발송 API 연동
* **알림 대상:** 새 맞춤 혜택, 마감 임박 혜택, 저장 혜택 변경, 추가 정보 요청
* **추후 확장:** 카카오 알림톡은 MVP 이후 별도 연동 항목으로 둔다.

## 4. 데이터 수집 및 정규화

### 4.1. 데이터 소스

* 공공데이터포털의 복지 관련 Open API
* 정부24/보조금24 계열 공공서비스 정보
* 복지로 및 한국사회보장정보원 복지서비스 정보
* 지자체 복지서비스 API 또는 공개 데이터

### 4.2. 수집 방식

* 매일 새벽 1회 이상 정기 수집한다.
* 사용자 검색 시점에 외부 API를 직접 호출하지 않는다.
* API 응답 원문은 원본 보관 테이블에 저장하고, 서비스용 필드는 별도 정규화 테이블에 저장한다.
* 동일 공고는 기관명, 서비스명, 신청 기간, 원문 URL, 공고 ID를 기준으로 중복 제거한다.
* API 변경에 대비해 데이터 소스별 어댑터를 분리한다.

### 4.3. 정규화 필드

복지 혜택 데이터는 최소 다음 필드를 가진다.

* 서비스 ID
* 서비스명
* 제공 기관
* 지역: 전국, 시/도, 시/군/구
* 대상 연령
* 대상 가구 유형
* 대상 상태: 구직, 재직, 학생, 소상공인, 장애, 임신/출산, 노인 등
* 소득 조건
* 주거 조건
* 지원 내용
* 지원 금액 또는 지원 형태
* 신청 시작일
* 신청 마감일
* 신청 방법
* 신청 URL
* 문의처
* 원문
* 쉬운 말 요약
* 추천 태그
* 마지막 수집 시각

## 5. 사용자 데이터 모델

### 5.1. 사용자 프로필

사용자 프로필은 맞춤 추천을 위해 다음 구조로 저장한다.

* 사용자 ID
* 출생년도 또는 연령대
* 거주지 시/도
* 거주지 시/군/구
* 현재 상태
* 가구원 수
* 가구 구성
* 소득 구간
* 주거 형태
* 고용 상태 상세
* 관심 분야
* 특수 상황
* 알림 수신 채널
* 알림 수신 동의 상태
* 개인정보 처리 동의 상태
* 마지막 프로필 수정 시각

### 5.2. 민감정보 처리 원칙

* 정확한 주민등록번호, 금융정보, 건강정보 원문은 MVP에서 수집하지 않는다.
* 소득은 정확한 금액보다 구간형 선택지를 우선 사용한다.
* 장애, 질병, 한부모 등 민감할 수 있는 항목은 선택 입력으로 처리한다.
* 맞춤 추천에 필요한 최소 정보만 저장한다.
* 사용자가 언제든 프로필 정보와 알림 동의를 수정하거나 삭제할 수 있어야 한다.

## 6. 추천 및 RAG 설계

### 6.1. 추천 흐름

1. 사용자가 자연어로 상황을 입력한다.
2. AI가 사용자 입력에서 의도, 생활 상황, 필요한 지원 분야를 추출한다.
3. 시스템이 사용자 프로필과 추출된 의도를 기준으로 구조화 필터를 적용한다.
4. 필터링된 후보군 안에서 벡터 검색으로 의미적으로 가까운 혜택을 찾는다.
5. 추천 후보를 점수화하고 "가능성 높음 / 확인 필요 / 가능성 낮음"으로 분류한다.
6. AI가 후보 혜택을 쉬운 말로 설명한다.
7. 확정 판단이 필요한 조건은 추가 질문으로 사용자에게 묻는다.

### 6.2. 추천 점수 기준

추천 점수는 다음 요소를 조합한다.

* 지역 일치 여부
* 연령 조건 일치 여부
* 현재 상태 일치 여부
* 가구/주거/소득 조건 일치 가능성
* 신청 기간 유효 여부
* 사용자 질문과 공고 내용의 벡터 유사도
* 사용자의 관심 분야 일치 여부

### 6.3. 응답 제한

* AI는 "반드시 받을 수 있다"라고 말하지 않는다.
* AI는 "신청 가능성이 높다", "조건 확인이 필요하다"처럼 안전한 표현을 사용한다.
* 원문에 없는 금액, 기한, 자격 조건을 생성하지 않는다.
* 중요한 정보는 원문 필드 또는 검수된 요약 필드에서만 가져온다.

## 7. 쉬운 말 요약 설계

### 7.1. 요약 생성 시점

* 복지 공고 수집 후 백그라운드 작업으로 요약을 생성한다.
* 사용자가 상세 화면을 열 때마다 실시간으로 요약하지 않는다.
* 요약 실패 시 원문과 기본 정규화 필드를 우선 표시한다.

### 7.2. 요약 결과 구조

각 공고는 다음 요약 필드를 가진다.

* 한 줄 요약
* 받을 수 있는 지원
* 신청 대상
* 신청 조건
* 신청 기간
* 신청 방법
* 준비 서류
* 어려운 용어 풀이
* 주의할 점

### 7.3. 검수

* 관리자 화면에서 AI 요약을 수정할 수 있어야 한다.
* 수정된 요약은 AI 재생성 결과보다 우선한다.
* 사용자에게 노출된 요약의 생성 시각과 원문 기준일을 저장한다.

## 8. 문자/이메일 알림 설계 (부수적 옵션)

본 서비스의 핵심 가치는 화면을 통해 복지 정보를 직관적으로 시각화하여 전달하는 것이며, 문자/이메일 알림은 사용자가 원할 때 활성화하는 보조적인 편의 기능이다.

### 8.1. 알림 트리거

알림은 다음 조건에서 생성한다.

* 사용자 조건에 맞는 새 복지 혜택이 수집된 경우
* 저장한 혜택의 신청 마감일이 가까운 경우
* 저장한 혜택의 신청 기간 또는 내용이 변경된 경우
* 사용자 프로필을 보완하면 더 정확히 판단할 수 있는 혜택이 있는 경우

### 8.2. 알림 발송 정책

* 동일 혜택에 대해 같은 사용자에게 중복 알림을 반복 발송하지 않는다.
* 사용자가 설정한 관심 분야와 알림 빈도를 반영한다.
* 야간 발송을 피하고, 기본 발송 시간은 오전 또는 오후 업무 시간대로 제한한다.
* 문자에는 짧은 설명과 상세 링크를 포함한다.
* 이메일에는 추천 이유, 신청 기한, 준비 서류, 상세 링크를 포함한다.

### 8.3. 알림 테이블

알림 발송 이력은 다음 정보를 저장한다.

* 알림 ID
* 사용자 ID
* 혜택 ID
* 채널: SMS 또는 Email
* 알림 유형
* 제목
* 본문
* 발송 상태: 대기, 성공, 실패, 수신 거부
* 발송 요청 시각
* 발송 완료 시각
* 실패 사유

## 9. 주요 화면 요구사항

### 9.1. 온보딩

* 사용자가 부담을 느끼지 않도록 기본 정보부터 단계적으로 입력받는다.
* 민감한 선택 항목은 "건너뛰기"를 제공한다.
* 입력한 정보가 왜 필요한지 짧게 설명한다.

### 9.2. 상담 화면

* 채팅 UI를 제공한다.
* 답변은 긴 문단보다 혜택 카드와 짧은 설명 중심으로 구성한다.
* 추가 질문은 한 번에 하나씩만 한다.

### 9.3. 혜택 상세 화면

* 쉬운 말 요약을 최상단에 보여준다.
* 원문 보기 버튼을 제공한다.
* 신청 링크, 문의처, 준비 서류 체크리스트를 제공한다.

### 9.4. 알림 설정

* 문자 수신 여부
* 이메일 수신 여부
* 관심 분야
* 알림 빈도
* 수신 거부

### 9.5. 관리자 화면

* 수집 공고 목록
* AI 요약 검수
* 추천 태그 수정
* 알림 발송 상태 확인
* 수집 실패 로그 확인

## 10. 보안 및 개인정보

* 사용자 비밀번호는 직접 저장하지 않고 인증 제공자의 안전한 인증 방식을 사용한다.
* 개인정보와 민감정보는 DB 암호화 또는 필드 암호화 대상이다.
* 알림 발송용 전화번호와 이메일은 별도 동의 상태와 함께 관리한다.
* 개인정보 삭제 요청 시 프로필, 알림 설정, 알림 이력을 정책에 따라 삭제 또는 비식별화한다.
* 관리자 접근은 역할 기반 권한으로 제한한다.
* 운영 로그에 전화번호, 이메일, 민감한 상담 원문이 그대로 남지 않도록 마스킹한다.

## 11. 장애 및 예외 처리

* 공공 API 수집 실패 시 이전 수집 데이터를 계속 사용한다.
* AI 요약 실패 시 원문과 기본 필드를 표시하고 재시도 큐에 넣는다.
* 문자/이메일 발송 실패 시 실패 사유를 저장하고 제한된 횟수만 재시도한다.
* 벡터 검색 장애 시 구조화 필터 기반 추천으로 대체한다.
* 사용자의 조건이 부족하면 무리하게 추천하지 않고 필요한 추가 질문을 한다.

## 12. 성능 목표

* 일반 상담 응답: 3초 이내 목표
* 혜택 상세 화면 로딩: 1초 이내 목표
* 데이터 수집 작업: 사용자 요청과 분리된 백그라운드 작업으로 실행
* 알림 매칭 작업: 매일 1회 이상 실행
* 외부 API 장애 시 사용자 화면 장애로 이어지지 않도록 캐시된 내부 데이터를 사용

## 13. MVP 구현 순서

1. Next.js PWA 프로젝트 구성
2. PostgreSQL 및 기본 데이터 모델 구성
3. 사용자 인증과 확장 프로필 입력 구현
4. 복지 공고 수집기와 정규화 파이프라인 구현
5. 쉬운 말 요약 생성 및 저장
6. 자연어 상담 API와 추천 로직 구현
7. 혜택 카드/상세/저장 화면 구현
8. 문자/이메일 알림 발송 워커 구현
9. 관리자 검수 화면 구현
10. 접근성, 보안, 알림 동의, 장애 처리 점검
